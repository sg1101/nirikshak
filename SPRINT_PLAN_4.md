## Task Breakdown

### Phase 1: Audit Replay — The Audit Drill (Day 9, first half)

**T4.1 — Replay engine (`audit/replay.py`)**

The demo segment 5 centerpiece: "Why did Bidder X fail criterion Y?"

Function: `replay_verdict(entry_hash: str, session) -> ReplayResult`

Steps:
1. Fetch the audit entry by hash
2. Extract `bidder_id` and `criterion_id` from the entry payload
3. Follow the chain backward to reconstruct the full action history for that bidder+criterion:
   - Find `criteria_locked` entry → get the locked CriteriaSpec content_hash
   - Find `bidder_ingested` entry → get the bidder's document content hashes
   - Find `evidence_extracted` entry → get the original evidence
   - Find `rule_fired` entry → get the historical verdict
4. Re-load the locked CriteriaSpec from DB (by content_hash match)
5. Re-load the bidder's evidence claims from DB
6. Re-run the verdict rule on the evidence
7. Compare recomputed verdict state vs historical verdict state
8. Return `ReplayResult` with: match (bool), historical verdict, recomputed verdict, full chain of events, divergence explanation if different

Data model:
```python
class ReplayResult(BaseModel):
    entry_hash: str
    bidder_id: str
    criterion_id: str
    historical_state: str
    recomputed_state: str
    match: bool
    chain_of_events: list[dict]     # simplified audit entries in order
    divergence_reason: str | None   # only if match is False
```

API endpoint: `POST /api/audit/replay` with body `{"entry_hash": "..."}`

**Acceptance:** Replay on any `rule_fired` audit entry → recomputed verdict matches historical. Replay returns the full chain of events.

### Phase 2: Signed PDF Report (Day 9, second half)

**T4.3 — PDF report generator (`audit/signer.py`)**

Generate a consolidated evaluation report as a PDF using ReportLab.

Function: `generate_report(tender_id: UUID, session) -> bytes`

Report contents:
1. **Cover page**: Tender title, authority, evaluation date, Nirikshak logo/header
2. **Tender summary**: Title, estimated value, bid date, criteria spec hash
3. **Criteria summary table**: ID, type, description, mandatory flag
4. **Per-bidder verdict cards** (one section per bidder):
   - Bidder name, aggregate verdict (bold colored)
   - Per-criterion table: criterion ID, verdict state, rule fired, reason (first 100 chars)
   - Evidence summary: claim count, verified count, avg confidence
5. **Audit log summary**: Entry count, chain verification status, first and last entry hashes
6. **Self-signed digital signature block**:
   - Generate a self-signed certificate (using `cryptography` library or simple HMAC)
   - Sign the report content hash
   - Include: "This report was digitally signed using a self-generated key. Production deployment would use a CCA-licensed Digital Signature Certificate (DSC)."
7. **Footer**: "Generated by Nirikshak | Prototype — Not for official use"

API endpoint: `GET /api/tenders/{tender_id}/report` → returns PDF bytes

**Acceptance:** Download a PDF with all bidder verdicts, criteria summary, audit hash, signature block.
